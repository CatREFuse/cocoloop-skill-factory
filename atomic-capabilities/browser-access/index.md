# 浏览器访问

## 适用场景

这个能力用于打开页面、读取页面信息、确认页面状态、完成简单交互，以及把网页内容转成后续可用的资料。

## 选择条件

- 目标信息只在网页里。
- 需要确认界面状态或渲染结果。
- 需要做页面级交互或表单检查。
- 需要对外部网页进行可见性验证。

## 输入

- 目标 URL
- 页面目标
- 预期状态
- 是否需要交互
- 登录或权限前提

## 输出

- 页面可见信息
- 提取到的内容
- 交互结果
- 页面状态判断

## 最小 SOP

### 1. 先判断是不是必须上浏览器

- 文本接口或公开 API 足够时，不上浏览器
- 只有网页能拿到目标内容时，再上浏览器

### 2. 再选执行方式

- 只读检查：优先静态抓取或轻量页面读取
- 表单或按钮交互：优先 Playwright 类工具
- 长链路登录：拆成独立能力，不和主流程混写

### 3. 写清前置条件

- 目标 URL
- 登录状态
- 是否需要 cookie
- 失败时的人工确认方式

### 4. 保留验证痕迹

至少保留一种：

- 提取到的结构化文本
- 截图
- 页面状态摘要

## 实操建议

- 对高风控页面，不把浏览器自动化当成默认成功路径
- 对抓取类任务，优先保留 API 或 fixture fallback
- 在最终 Skill 里把“浏览器验证”和“核心抓取逻辑”拆开，避免一个失败拖垮全部流程

## 边界

- 不把复杂登录、强风控和频繁变化的界面当成稳定前提。
- 不默认可以绕过权限。
- 不把浏览器当成万能抓取器，文本层能做的先文本层处理。

## 降级策略

- 页面打不开时，退回到网页摘要、手动输入或文档参考。
- 如果交互失败，只保留可见内容和失败原因。
- 如果页面变化太快，改为截图核对或人工确认。

## 与主流程的关系

这个能力适合在调研、验证和构建检查时使用，尤其适合需要确认真实页面行为的场景。
